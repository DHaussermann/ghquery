# How It Works

## Overview

`ghquery` is a Go tool that queries GitHub for pull requests, analyzes each one for risk using isolated Claude subagents, and delivers a structured report to a channel. It can be run as a **web UI**, a **CLI**, or a **scheduled job**.

```

│          Entry points: Web UI  ·  CLI  ·  Scheduled job           │
└───────────────────────────────────────────────────────────────────┘
```

## The Three Phases

### Phase 1 — Fetch

Queries the GitHub Pull Requests API filtered by author and date range, then fetches the **aggregate PR diff** — the net result of all commits, equivalent to GitHub's "Files Changed" tab.

Go pre-computes the **blast radius** for each PR before sending to analysis:
- `files_changed` — number of files in the PR
- `dirs_changed` — unique directories touched
- `areas_affected` — classified as server, webapp, mobile, tests, plugin, infra, database
- `total_lines` — additions + deletions
- `cross_area` — whether the PR spans multiple system areas

**Two fetch modes:**
- **Query mode** — fetch all matching PRs for the configured authors/repos/days window
- **Single-PR mode** — paste any GitHub PR URL into the sidebar input to bypass filters and analyze exactly one PR

**API usage:**
- Works unauthenticated for public repos (60 req/hr)
- Set `github_token` in config.yaml for 5,000 req/hr
- API call count and rate limit remaining are logged after every run

#### CodeRabbit parsing (optional)

When **Use CodeRabbit** is enabled, each PR's body is scanned for CodeRabbit's "Change Impact" section — a block injected between HTML comment markers that contains a tier, regression risk paragraph, and QA recommendation.

```
<!-- This is an auto-generated comment: release notes by coderabbit.ai -->
## Change Impact: 🟢 Low

**Regression Risk:** Minimal. ...
**QA Recommendation:** Manual QA is optional. ...

*Generated by CodeRabbitAI*
<!-- end of auto-generated comment: release notes by coderabbit.ai -->
```

Two comment shapes are handled: one with an italicized `*Generated by CodeRabbitAI*` footer and one without (just the HTML comment delimiters). Any body containing `coderabbit` and a parseable `Change Impact:` line is accepted.

Parsing is free — it reads from `pr.body`, which the GitHub API already returns, so no extra API call is needed.

### Phase 2 — Analysis

#### CodeRabbit shortcut

For any PR that has parsed CodeRabbit data (and the toggle is on), **Pass A and Pass B are skipped entirely**. The CodeRabbit tier, regression paragraph, and QA recommendation are used directly to populate the `PRRisk` row, and the row is flagged `source: "coderabbit"`. No Claude CLI call is made for that PR.

PRs without CodeRabbit data fall through to the two-pass agent flow.

Log line: `[analyze] N PRs total — M sourced from CodeRabbit, K to analyze by agent`

#### Two-Pass Agent Flow

Each remaining PR is analyzed in **complete isolation** by its own Claude subagent (no context pollution between PRs). Two sequential passes run per PR:

**Pass A — Branch Enumeration (60s timeout)**

A short, constrained Claude call with a single job: list every new code branch introduced by the diff and what each test addition covers. No scoring, no recommendations, no commentary.

Output is structured JSON:
```json
{
  "new_branches": [
    { "path": "server/foo/bar.go", "branch": "statfs timeout-fired path" }
  ],
  "test_changes": [
    { "path": "server/foo/bar_test.go", "covers": "linux happy path" }
  ]
}
```

This output is handed to Pass B as `PRE-COMPUTED BRANCH ENUMERATION (use this as your AUTHORITATIVE list)`. The constrained scope keeps Pass A resistant to the model's training prior that "if tests exist, coverage is fine" — it must enumerate granularly instead.

**Pass B — Risk Scoring (90s timeout)**

The full risk-analyzer skill, fed with Pass A's branch list as pre-computed context. The agent scores 6 dimensions and derives a risk tier. The branch list anchors the coverage analysis step, preventing the scoring agent from collapsing fine-grained signals into coarse labels.

If Pass A fails or times out, Pass B still runs without the branch context (degraded but functional).

**Concurrency:** 3 PRs analyzed in parallel (each doing its own Pass A → Pass B sequentially). Agents are named alpha / beta / gamma in logs.

#### What each analysis returns

- `risk_level` — HIGH / MEDIUM / LOW / UNKNOWN
- `risk_score` — 0.0–10.0 (N/A for CodeRabbit-sourced rows)
- `dimensions` — 6 dimension scores (N/A for CodeRabbit-sourced rows)
- `risk_reason` — specific observations from the diff
- `qa_recommendations` — up to 3 concrete user flows to verify
- `source` — `""` (agent) or `"coderabbit"`

### Phase 3 — Output

**Web UI** — the primary interactive surface. Sortable risk table, click-to-scroll drilldowns, Untested / Reviewed split, CodeRabbit `[CR]` badges, "Send to Hook" button to push the report to the channel.

**webhook** — formatted Markdown post. Untested work (merged with no QA approval) surfaces first. Drilldown sections follow for HIGH then MEDIUM PRs in each group. CodeRabbit-sourced rows are marked `(CR)` with a footer `_Generated by CodeRabbitAI_`. If delivery fails, the report is auto-saved to `~/.ghquery/reports/`.

**Scheduled runs** — via `ghquery schedule install`, runs on macOS launchd (weekdays / daily / weekly) and posts to the channel automatically at the configured time.

## Risk Scoring

Each agent-analyzed PR receives a categorical level and a numeric score across 6 dimensions:

| Dimension | What it measures |
|-----------|-----------------|
| **blast_radius** | Files, directories, and system areas affected (pre-computed by Go) |
| **complexity** | Logic density — nested conditionals, concurrency, state mutations |
| **regression_surface** | Fragility of the touched area — shared utilities, core libraries, untested branches |
| **data_integrity** | Risk of data corruption — migrations, query changes, ORM modifications |
| **security_surface** | Auth, input validation, API exposure, secret handling |
| **infra_config** | CI/CD, deployment manifests, dependency upgrades |

`data_integrity` and `security_surface` carry more weight than other dimensions. A score ≥ 7.0 → HIGH; 4.0–6.9 → MEDIUM; < 4.0 → LOW. Hard overrides can force HIGH regardless of numeric score (schema migrations, auth changes, removal of security validation, etc.).

---

## Untested vs. Reviewed

If a `qa` team is configured, merged PRs are classified at report time:

- **Untested** — merged but no member of the QA team has approved the PR
- **Reviewed** — merged with at least one QA team approval

Both the web UI and webhook output split these into separate sections, with Untested surfaced first as higher priority.

## Configuration

`config.yaml` structure:

```yaml
catalog:
  repos:
    - org/repo-one
  teams:
    qa:         [user1, user2]
    reliability: [user3, user4]
  author_names:
    member-1: "Harrison Healey"

query:
  repos:        [org/repo-one]
  authors:      [member-1]
  days:         7
  mode:         untested        # "all" or "untested"
  skip_analysis: false

schedule:
  enabled:    true
  frequency:  weekdays          # "weekdays", "daily", "weekly"
  time:       "08:00"
  tz:         America/Toronto

github_token: "ghp_..."
webhook_url:  "https://..."
```

## Tuning the Risk Analysis

The risk criteria live in `.claude/agents/risk-analyzer.md` — a plain Markdown file editable without recompilation. Key sections:

- **`## Risk Classification`** — categorical level derivation, hard overrides, tightened triggers, LOW/MEDIUM pattern lists
- **`## Analysis Process`** — the numbered steps agents follow, including the forward-looking failure analysis (Step 5) that anchors Pass B's coverage check on Pass A's branch list
- **`## Output Format`** — fields returned in the JSON response

---

## Safety Guards

- **Max 14-day lookback** — prevents unbounded queries
- **Days always defaults to 7** — can never be zero or negative
- **Rate limit monitoring** — warns when GitHub rate limit is low
- **Per-pass timeouts** — Pass A: 60s, Pass B: 90s per PR
- **Failure isolation** — one failed PR analysis doesn't block the rest; failures are marked UNKNOWN
- **Pass A degrades gracefully** — if Pass A times out, Pass B still runs without the branch context
- **CodeRabbit falls through** — if a PR has no parseable CodeRabbit data, the agent runs normally
- **Webhook failover** — if webhook delivery fails, the report is saved to `~/.ghquery/reports/`
